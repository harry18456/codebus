Sanity check · cross-cutting decisions

C1 · +1 step typography

Approve as-is. The 13/11/10 scale was tuned at macOS Retina 2x — that doesn't survive Windows 100% scaling, and CJK glyph hinting at sub-13px is genuinely soft (zero subjective tolerance from me on this; the audit instinct is correct). Bumping headings together preserves rhythm.

One nudge: when you re-snapshot at 1920×1080 Win 100%, also spot-check 4K @ 150% scaling and a macOS retina baseline. If macOS feels chunky after the bump, that's the signal you need the user-level font-scale setting later — but agree with not adding it yet.

C2 · i18n holes

Aligned. Cat A / B / C is a clean accounting and the reserved-English list is exactly the right discipline (jargon users learn = stays English; everything else translates). Two small refinements:

  • For Cat C aria-labels, prefer one shared key per concept (one a11y.pageNotFound, not three title="Page not found") — easier to audit, easier to translate.
  • "Endpoint configuration is incomplete:" — translate as "端點設定不完整：" with 端點 in Chinese. The CLI-aligned jargon is the key names (base_url, api_version), not the sentence around them.

C3 · CJK section labels

This is the biggest miss in the v1 spec and I owe you a real answer (see Q1 below for the recommendation + visuals). Short form: drop uppercase-tracked entirely in zh-tw. It's a Latin-only affordance. Substitute amber-left-bar in fg-secondary at meta size. Works for both scripts.

C4 · Border hairlines

Approve. #1f1f1f on #0a0a0a is below ClearType's perceptual floor — that's not a calibration thing, that's just thin sub-pixel anti-aliasing losing the line. Promote default to --border-strong: #2a2a2a. Keep the #1f1f1f token only for in-card row separators where you genuinely want "almost invisible". Spec was written looking at Figma against pure black, where any line wins.

C5 · Three-state status

Yes — absorb into the spec. Interrupted is a meaningful, recoverable state and bundling it into failed was a v1 oversight. Lock the tokens as you have them:

canonical three-state, applied to a goals row
auth flow 怎麼運作 14m ago
列出這 repo 的主要入口跟模組 interrupted
extract billing webhook contract failed
map the public API surface analyzing token validation flow| streaming · 4.2k tok

Running uses the same amber as interrupted but with an outer pulse ring + caret — those are motion/affordance, not color. Don't try to find a fourth hue.

C6 · ChatWidget vs Cancel collision

Aligned. Cancel belongs in the title row right — that's already in the v1 spec, the bottom-right Cancel was an implementation regression. Fixing W3 fixes C6 for free.

Bigger contract: the bottom-right is ChatWidget's territory and nothing else docks there. Any future "ambient goal-progress" affordance lives on the ChatWidget bubble itself (e.g. amber pulse dot when a goal is running), not as a sibling button. This keeps the visual edge of the app uncluttered and gives the widget a real signaling role.

C7 · CTA placement

Aligned. + New goal / + New quiz belong in the content header row paired with a screen <h1>. The topbar is drag-only. Don't add an <h1> in the topbar to "save space" — that breaks the affordance.

C8 · Codex shell wrapper

Aligned with both moves: client-side wrapper extraction in ActivityStreamItem for the readability fix, plus the upstream UTF-8 issue (which is genuinely a codex-side bug, not yours to fix in UI). Worth filing the codex bug separately; the LLM retrying on its own is real token waste.

Cut 05 · Cmd+K Overlay

Right call. v1 split the surface because the team hadn't built ChatWidget yet; now that ChatWidget exists with multi-turn + promote-to-goal + token counter, maintaining two chat-like surfaces is pure overhead. Keep ⌘K as the ChatWidget toggle. The escape hatch (centered-modal mode on the same widget) is the right shape if spotlight UX ever comes back as a need — don't build it speculatively.

Vault terminology removal

Right call. Two names for one container = friction, and "近期 VAULT" was visually broken anyway. Keep VaultEntry / codebus init / docs — the internal model name is fine.

recommend amber left-bar

Drop uppercase-tracked entirely in zh-tw. The treatment is a Latin-script ceremony — caps + Roman tracking creates "officialness". CJK has no equivalent mechanism at that size. Trying to retrofit either looks broken (mixed strings) or just reads "small bold gray".

The replacement should signal "this starts a group" without trying to be ceremonial:

Final call:

  • Use C (amber left-bar) as the universal section-label treatment. Apply it everywhere a label is needed in any locale (Latin or CJK).
  • For the wiki taxonomy headers (MODULES, PROCESSES, SYNTHESIS) — keep them English-uppercase as bucket names (they're domain identifiers from the spec, not user-translatable), but render with the amber-bar treatment, not standalone caps+tracking.
  • Drop the label entirely (option B) only where there's exactly one group — the Workspace nav is the only such place.
  • Retire the uppercase-tracked CSS token. It was wrong for zh, and it's now redundant for en.
02 02c Interrupted — into the spec, or codebus extension?
absorb into spec

Interrupted vs Failed is a real UX distinction, not codebus-specific — any long-running LLM job has a "you can retry safely" and "something is broken, retrying won't help" axis. v1 spec collapsed them because the mocks were paper-only and never confronted the actual lifecycle. Your three-state is the canonical version going forward.

Two things to tighten while you're formalizing:

  • Name the routing rule explicitly in the spec — user-cancel / app-close / network-drop = interrupted; LLM API hard error / sandbox crash / lint loop irrecoverable = failed. If you can't tell which side an error falls on, default to interrupted (retry is cheaper than investigation).
  • The component is named RunDetailCancelled but handles all three interrupt sources. Rename to RunDetailInterrupted so the file matches the state name in the spec. Small thing, but state-vs-trigger naming drift is a slow source of bugs.

Banner copy: I'd nudge your draft from "app 中途被關閉" → "中途離開了 codebus，wiki 還沒寫完。可以重跑這個 goal 補上。" — "被關閉" sounds blameful (was it me? did codebus crash?), and "可以重跑" is more active than "需要的話". Same shape, different mood.

03 Quiz wizard form factor — fullscreen view OK?
accept the departure

Yes — fullscreen wizard is the right shape and the parallel with + New goal was the original mistake. Goal flow is one-shot (modal is fine); quiz flow is six steps with a human-in-the-loop scope confirmation (modal is wrong). Different shapes get different patterns.

One ask while you're refactoring this:

  • When the wizard is active, the topbar + New quiz should be hidden, not disabled. Disabled creates "what happens if I click that?" hesitation; hiding is unambiguous. (Same rule as "don't show + New Goal during a Goal Running view".)
  • The scope-confirmation step is the single most valuable thing in this flow — it's the only point where users can correct the LLM's page picks before paying for generation. Lavish design attention here: show the picked pages with their summaries (or first H2 line), let users uncheck individual pages, expose why these were picked (matched keywords, recently updated, recently authored by). This is the surface where users learn to trust the LLM's scoping.
  • Wizard exits → return to Quiz history. Don't return to the empty state or Lobby. Continuity matters here.
04 Step indicator for the Quiz wizard
dots + label

Text alone ("Step 2/4 · Scope") communicates position but not progression. Progress dots alone look like a generic onboarding flow and don't say what each step is. Both, side-by-side in the header strip:

wizard header — dots + step label
Step 2 / 4 · Scope
Step 4 / 4 · Answering · Q3 / 5

Dots: 7px circles, filled-amber for current, filled-tertiary for completed, ring-only for pending. Sits flush-left in the header strip where Quiz: <scope> normally goes. During the answering step, the dot indicator drops away and the strip shows Q3 / 5 instead (because per-question progress is the live concern, not per-step).

Concretely the four steps are: topic → scope → generating → answering. Don't break "answering" into per-question dots — that's QuizAnswering's job.

05 Bus motion vocabulary
lock 2 moods, nothing else

Yes to the distinction. Codify it as a 2-mood vocabulary and stop there — any new bus animation proposal has to map to one of these or get rejected:

  • Moving forward — codebus-bus-roll (translate -26→12px, ±2° rotate, slight Y bob, 1.8s). Used for: LoadingOverlay (vault init). Conceptually: "codebus is doing work that will end".
  • Idling in place — vertical 2px bob, 1px horizontal jitter, 1.4s, no rotation. Used for: 04b Lobby empty hero. Conceptually: "codebus is waiting for you".

Hard nos:

  • No bus animation on goal-running. That surface already has the amber pulsing dot and the stream-tail caret — adding bus motion competes with them. The pulsing dot is the "codebus is currently driving" affordance.
  • No bus animation on Quiz generation. Same reasoning. (And we'd just be reusing LoadingOverlay's mood there, which would dilute it.)
  • Wordmark 🚌 stays static. Static glyph + 2 deliberate motion moments is the right cadence. Three+ animated buses tips into mascot-overuse.

Both animations gate on prefers-reduced-motion: reduce — fall back to static 🚌 with no transform.

09 "Goal Log" → "旅行日誌"
yes

Perfect. The bus metaphor pays off when it shows up in deliberate, low-frequency places: wordmark, LoadingOverlay, the 04b idle hero — and now "旅行日誌". You earn warmth without spending it.

I considered "公車日誌" briefly because it's more direct, but it's also more literal and steals user attention. "旅行日誌" sits one layer abstract — it's the user's trip through the codebase, recorded by codebus. That framing matches what the file actually contains better than the engineering framing does.

10 Page metadata bar on wiki preview
yes — single line, provenance only

Yes, and keep it tight. Position: directly below the page title, above the body content. Single line, mono-12 (post-bump), fg-tertiary. Format:

wiki page header (after the title)

桌面工作台

Last updated by 完整研究專案目的 · 12h ago · 4 sources

Constraints:

  • Provenance only. Goal name → time → source count. Resist adding word count, tag chips, view count, or anything analytics-flavored. Provenance is what users actually need to judge "do I trust this page?" — the rest is noise.
  • Goal name hovers + clicks through to that goal's detail view. This is the closing of the loop — "this page exists because this goal ran" is the single most important relationship in codebus, and one click is the price.
  • "4 sources" is the wikilink-citation count from the page body. Only show if > 0.
  • If the page was edited via a follow-up goal, list the most recent. Don't try to show authorship history here — that belongs in 旅行日誌.

Related: WP5 (the "humans edit by running a goal" philosophy) — I'd add a quiet hint at the page footer, not the metadata bar. Something like "想改這頁？跑一個 goal 跟 codebus 說該怎麼改 →" as a fg-tertiary link that opens NewGoalModal pre-filled with "修改 [此頁] 的 …". Keeps the philosophy visible without nagging.

Surprises & pushback

Things in the audit that surprised me (positively)

  • The scope-confirmation step in the Quiz wizard is genuinely good UX I would have spec'd in v1 if I'd thought through the flow shape. Don't bury it as "an extra step" — it's the single most useful trust-builder in the entire app. Treat it as a flagship surface.
  • The bus-roll keyframes for LoadingOverlay. Translate + bob + rotate is exactly the playful-but-not-too-much I'd want, and I haven't seen it before — please keep the file checked in even if you redesign other parts later. It's a brand asset.
  • The "interrupted" state. I should have caught this in v1. The fact that you built it before asking, and the routing rules came out of real failure modes (app close / cancel / network), means it's grounded. Add it to the spec.
  • QL1 (hash ID as quiz row title) is a clean catch. "Show users the topic they typed, not the topic's internal ID" is the kind of bug that's invisible to engineers and obvious to users. Prioritize this.

Things I'd push back on

R7-1 · "Replace 💬 with lucide MessageSquare"

I'll push back gently. The audit says 💬 is "the only cartoon element left" — but consider that's a feature: codebus is trying to be warm. The 🚌 wordmark, the LoadingOverlay bus, the idle hero, the 旅行日誌 — all the warmth lives in a handful of deliberate emoji moments. If 💬 is the chat one, that's coherent.

Two alternatives before you reach for MessageSquare:

  1. Keep 💬 as-is. It's exactly one emoji on exactly one always-visible affordance. It's only "cartoony" relative to Linear-tight; in the context of a brand that already has 🚌 in the wordmark, it lands fine.
  2. Replace 💬 with 🚌 specifically — ChatWidget is a chat with codebus-the-bus, the bus icon doubles as a "talk to your driver" affordance. Tight fit with brand. Caveat: now 🚌 would appear in 4 places (wordmark, loading, idle hero, chat bubble); risks dilution.

If the team's gut is still "MessageSquare", do it — but make sure the brand has somewhere else to breathe. Don't strip all the warmth out chasing Linear consistency. Codebus isn't Linear, it's a friendlier sibling.

D2 · "注意事項" / Follow-ups section

Keep it, but be ruthless about the empty case. The audit's verification plan (5/5 different goal types should yield notes) is the right test. If any goal type comes back with no notes, the section needs to fully hide — not render as "Follow-ups: (empty)". Empty sections train users to skip the section, which kills its value when it does have something to say.

Visual grouping in 02 activity stream — don't over-engineer it

Your hybrid plan (keep emoji banners + add tool-kind grouping with 1px-left-border guide) is correct in shape, but resist the temptation to make every tool-kind cluster its own collapsible section. The spec's READING CODEBASE / WRITING WIKI split is two semantic phases of work (intake → output), not five tool flavors. Mirror that semantic split, not the technical taxonomy:

  • One implicit grouping by phase: reading-cluster (Read/Glob/Grep/Shell-for-inspection), writing-cluster (Write/Edit/Shell-for-mutation).
  • Within a cluster, individual tool calls stay flat — that's where the brand emoji banners do their work.
  • Raw event log stays in the collapsible stream-log card (default closed, per spec). That's where the engineering audience goes when they need the full firehose.

X2 · <CollapsibleStreamLog> shared component

Good abstraction call. One spec addition: the default-closed state should be sticky per-user, not per-page. If I expand the stream log on goal #1, opening goal #2 should remember that I'm a "show me the raw logs" person. Use localStorage; don't put it in the user-facing settings page yet.

GP8 · running row stream tail

Strongly approve. This is the v1-spec affordance I'm most attached to and the audit correctly flagged it as missing. The "current action narration" with amber blinking caret in the list view is what makes the Goals overview feel alive — without it, a running goal looks identical to a done goal, and users will check back constantly. Don't deprioritize this.
