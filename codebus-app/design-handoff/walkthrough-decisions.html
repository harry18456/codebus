1.5 · ⚠ 對齊點

0202 activity stream · 2-phase clustering + Shell kind enum

decision

Activity stream 走兩個 phase cluster：READING CODEBASE（intake）+ WRITING WIKI（output）。brand emoji banner 留 inline 在 phase 開頭。 Shell 工具靠 backend emit 的 kind: "read" | "inspect" | "mutation" 決定 cluster 歸屬——frontend 不做 regex 判斷。

2.1 · Shell kind enum 定義

Codebus-core emit Shell tool event 時帶一個額外欄位 kind，由 LLM intent layer 決定（你們已知道哪段 prompt 在發 shell call、它意圖讀還是寫）。三層分類：

kind 語意 典型範例 cluster 歸屬
"read" 讀檔案內容 / 列目錄 / pattern match cat / head / ls / find / rg / grep / wc READING CODEBASE
"inspect" 讀環境 / 中性 introspection · 沒有檔案內容 git status / git log / git diff / ps / env / which / npm ls READING CODEBASE
"mutation" 改 state（檔案 / git / 系統 / 安裝） rm / mv / mkdir / git commit / npm install / >file（redirect 寫入） WRITING WIKI

"inspect" 視覺上跟 "read" 同 cluster（都是 intake phase），但 tool icon prefix 不一樣， 後續若要做 per-tool filter / icon 也有區隔空間。LLM intent 拿不到時 default "inspect"（最安全的 unknown — 不歸到 mutation cluster， 才不會誤造成「咦它寫東西了？」的錯覺）。

2.2 · Tool kind → cluster 對應

連帶把所有 Claude API tool 一起定下歸屬：

toolclustericon prefix
ReadREADING CODEBASE📄
GlobREADING CODEBASE🗂
GrepREADING CODEBASE🔍
Shell · kind=readREADING CODEBASE$_
Shell · kind=inspectREADING CODEBASE$?
WriteWRITING WIKI✎
EditWRITING WIKI✎

Icon 用 mono ASCII 不用 emoji——這層是技術細節 ambient，不該再加 brand 量。Brand emoji 留給 phase banner（🚌 / 🎯 / 🤔）跟最後的 commit / done banner（🚏 / 🎉）。

2.4 · 細節規範

  • Brand emoji banner 不收進 cluster。 🚌 / 🎯 / 🤔 是「敘事層」、跨 cluster 流動， 不該被 cluster 的左 border guide 包進去。flat 排在 cluster 之間，當「段落之間的轉折」。
  • Cluster 之間可以重複出現（thought → reading → thought → reading → thought → writing）。 不要強制把所有 reading 集中成一個 cluster。LLM 會在思考之間穿插讀檔案、寫 wiki 之前重新讀一遍—— cluster 是「同類連續發生」的視覺收斂，不是 phase 的單例。
  • Live tail（W5）只出現在當前正在跑的 cluster 最底。 不要每個 cluster 都顯示 live tail——只有最新那一個 cluster 才有「spinner + amber 文字 + caret」。
  • Cluster collapsible · default open during run / closed when complete。 跑中保持展開（user 想看 ambient progress），跑完之後預設收合（02b 視圖該突顯成果，不是過程）。 CollapsibleStreamLog 一律 default closed（per-user sticky · key codebus.stream-log.default-open）。
  • Cluster count 不算 thought blocks。 "12 calls" 只算真的有 tool call 的 row。 thought blocks 是敘事、不是動作。

2.6 · ⚠ 對齊點

  • Shell kind 由 codebus-core 還是 codebus-quiz / codebus-goal 各 skill 自己決定？ 設計上偏好「skill 自己標」——goal skill 比較知道它發這個 git log 是為了 讀歷史（inspect）還是為了驗 commit message（也算 read）。core 統一 default 太粗。
  • 不在表內的 tool（Task / WebFetch / 未來新 tool）怎麼歸？ default 走 READING CODEBASE。emit 端可以額外帶 cluster: "reading" | "writing" override（這層才是 cluster identity， kind 是 cluster 內部分類）。如果你們覺得多一個欄位太冗，那就把 kind 擴成 "read" | "inspect" | "mutation" | "other-read" | "other-write"。
  • 02b done 的 cluster summary 文案。我寫了「12 reads · 195 shell · 6.2s」這種 mixed-language pattern——可以中文化（「讀檔 12 次 · shell 195 次 · 6.2 秒」）或繼續英文。 傾向英文短一點、ambient 用。但跟 i18n 原則有點摩擦，列出來對齊。

Reason tag 三種色

3.5 · Scope alert（兩版共用）

2-7 pages 是 quiz 的 sweet spot。超出範圍時顯示 amber alert，但不擋路：

scope · few 只挑到 1 頁，quiz 可能會太單薄。試試擴大主題、或先「重新規劃」看看 codebus 有沒有忽略相關頁面。
scope · many 挑到 9 頁，問題會很雜。建議 uncheck 一些次要的、或拆成多個 quiz 各自聚焦。

3.6 · State machine

scope-confirm step 的進出狀態：

觸發動作
進入此 step planning step 完成 emit scope-proposed 後到
「確認 · 開始出題」 送出 selected_paths: string[] · 進 generating step
「重新規劃」 退回 planning step · LLM 重新規劃（會在 prompt 帶 prior selection + uncheck 過的 path 當 negative signal）
「Cancel」 退回 Quiz history · 不存任何 state
uncheck 全部 page 「確認」disable · footer count 標紅 0 / 5 pages selected · pick at least 1
refresh / app 重開 state lost · 回 Quiz history（這步不持久化，比較像 modal 邊界）

3.7 · ⚠ 對齊點

  • 場 #3 直接決定走 A 還是 B。場 #3 我會問你們三件事：
    1. codebus-quiz prompt 內部目前怎麼挑 scope？是 LLM rank 還是 embedding similarity？
    2. 每頁的 metadata（authoring goal / last updated / linked from）是不是已經在 wiki frontmatter 裡？
    3. 實作 Design A 大概多少工？如果 ≥ 2-3 週、產品節奏吃緊就先上 B、A 列下個版本。
  • 「重新規劃」的 LLM context。我假設可以把 uncheck 過的 path 當 negative signal 餵回 prompt，這樣使用者第二次規劃會更貼題。如果你們 prompt 結構目前不接受這層 feedback, 「重新規劃」就會變成「再隨機挑一次」——體驗會差很多。場 #3 對。
  • Topic banner 的 emoji 用 🎓 還是 🎯。 Goal 已經用 🎯 當任務目標 banner； quiz 主題用 🎓 區分（同 sidebar Quiz tab 的 emoji）。如果你們覺得 🎯 跟 🎓 在 generation log 裡會搞混，可以兩個都用 🎓 統一 quiz brand。
