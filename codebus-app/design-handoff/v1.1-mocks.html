1.2 · 6 phase 對應 InitEvent

Backend 已在 codebus-core/src/vault/init.rs run_init emit 20+ 個 InitEvent。 UI 收斂成 6 階段 — 1:1 渲染太細、user 看不清。每階段對應一組 InitEvent （任一 event 進來 = 該階段亮起）。

# copy · zh-tw copy · en InitEvents 對應
1 準備車庫 Preparing garage Start · LayoutCreated · SourceGitignore

4 搭起 wiki 結構 Building wiki structure SchemaDone · ManifestDone · NavStubsDone · SkillBundlesDone · SettingsDone
5 註冊到 Obsidian Registering with Obsidian ObsidianResult · 可 skip · 見 1.3
6 上路前最後檢查 Final checks StarterConfigDone · CommitDone · Finished

1.3 · 階段語意

  • 進入新階段就更新副標、dot 點亮、bus 不重置。 bus 動畫是 ambient「車在跑」 訊號、不該被階段切換 reset；user 不該因為文字換了感受到突兀的視覺重啟。
  • 同一階段內多個 InitEvent 不重複切換副標。 一個階段只渲染一次它的副標、 停留到下個階段 event 進來。同階段 events 不可見的細節讓 backend logging 處理。
  • 階段 5 ObsidianResult skipped 時 dot 直接 done、不停留。 backend 不 emit ObsidianResult 也視為 done，但 UI 副標一閃即逝會像 bug — strategy：階段 5 進來時 emit minimum 300ms 停留（即使 backend 馬上跳階段 6 也撐住）， 讓 user 感受到「噢有這步、跳過了」而不是「咦剛剛閃了什麼」。
  • 所有 InitEvent 都完成（=Finished）後 overlay fade-out 200ms 收掉，不要 instant 消失（user 會 disorient）。

1.4 · 異常處理

  • 單一階段超過 20s 沒進到下一階段 → 副標下方加灰色 hint 行 「（這步比平常久一點，再等等…）」。對 user 心理預期管理。
  • InitEvent emit error → overlay 切「失敗模式」：bus 停下、step dot 變紅、 標題改「車子卡住了」、副標寫具體錯（從 LocalizedError 來），加 retry 按鈕。這個失敗 UI 跟 02c Interrupted banner 同樣 amber-warm 視覺（不是 hard-fail red）— init 失敗多半 是 disk / permission 之類可重試的事。
  • 無 backend 進度（fallback） → 若 backend 暫時還沒接 progress emit （roll-out 早期），UI 維持 v1 的靜態副標但開啟 bus 動畫；不 break。

1.5 · ⚠ 對齊點

align 階段 1 / 階段 6 是「過程性」的、可能太快閃過。如果實機跑下來這兩階段都 < 500ms 就過了、 閃 dot 沒意義 — 你們收到實機 timing 後 ping 我，可能要把首尾收進相鄰階段。

2.2 · 三種中斷原因 · 三個 banner 變體

從同一個 view 出，但 banner 文案根據 backend 提供的 interrupt_reason 換成對應變體。樣式都一樣（amber-tint banner），只有 heading label + 內文不同：

reason heading label banner 內文
"app-close" interrupt · app-close 中途離開了 codebus，wiki 還沒寫完。可以重跑這個 goal 補上。
"user-cancel" interrupt · user-cancel 你取消了這個 goal。已寫的 wiki 內容會保留，重跑會從頭開始。
"network-drop" interrupt · connection-lost codebus 中途失聯了。可以重跑這個 goal 試試看。
"unknown" (default) interrupt 這個 goal 沒跑完。可以重跑、或留著看記錄到哪。

Heading label 用 .mono 風格、半技術詞——是讓 user 知道 「codebus 知道自己怎麼斷的」這層 metadata，不是寫給每個 user 看的核心訊息。內文才是。

桌面工作台

Last updated by 完整研究專案目的 · 12h ago · 4 sources

桌面工作台是專案面向使用者的主模組。它不單純把 CLI 輸出包進一個 WebView，而是提供多分頁、多會話、側邊欄、設定、行動版 H5 入口、通知與更新檢查等完整操作殼層，讓 Claude Code 從一次性命令列操作變成長時間維持的開發工作區。

模組責任

這個模組負責三件事：

  • 啟動與恢復工作區狀態，例如設定、分頁、目前活躍會話。
  • 提供可視化操作面，例如側邊欄、分頁列、內容路由、行動版抽屜。
  • 透過 Tauri 原生層與 sidecar，接上真正的 server / CLI / adapters 執行面。

與其他模組的邊界

桌面工作台不直接執行 agent 推理，也不直接理解 IM 平台協議。它把會話與串流交給 session-backplane，把外部聊天平台交給 im-adapter-system。 這個邊界很重要，因為它讓 UI 可以繼續演進，例如增加 H5 行動端模式、通知導頁與設定頁，而不用把 bot SDK 或 CLI 細節塞進 React 元件樹。

4.2 · Metadata bar 規格

  • 位置：title 下方、body 上方。一行不換行（必要時 ellipsis）。
  • 字體：mono 12px、fg-tertiary。底線 1px solid --border，下方 14px gap。
  • 內容固定為三段： Last updated by <goal> · <time-ago> · <N> sources
  • Goal name 可點 → 跳該 goal 的 detail view（hover 加 amber dotted underline）。 這是 codebus 最核心的「page ← goal」反向關係。
  • sources count = body 內 wikilink 數量（不含 goal name 那個）。< 1 時整段不顯示。
  • 不加 tags / word count / view count / authors。 provenance only — 多加只會稀釋這條訊息的價值。

5.5 · 細節規範

  • 持久化：chat session 跟著 vault 走（per-vault chat history、不跨 vault）。 Mode 偏好不持久化 — 每次 vault open 都從 bubble idle 開始；⌘K 是 mode 切換的 fast-path。
  • Loading state：bot 回應途中 user message 下方加一個小 spinner row 「 想中…」（不另開 message bubble、 不佔太多視覺空間）。
  • Streaming response：bot message 邊收邊 render、結尾 blinking amber caret （沿用 02 stream tail caret）。
  • Error state：bot reply 渲染為 amber-tint error message（不用 red）— ChatWidget 錯誤幾乎都是 retryable（network drop、token limit、context too long），跟 Interrupted goal 同色語。
  • Promote-to-goal preview：點「↗ promote to goal」chip 不直接送出 goal， 而是開 NewGoalModal、prefill「描述你想搞懂的事」欄位為當前 user message。User 可以 改、確認、再送。Chat 對話本身 stays in ChatWidget（不消失）。
  • i18n Cat C 全解：aria-label 全走 i18n key （chat.open / chat.minimize / chat.expand-to-modal / chat.dock / chat.close）。

5.6 · ⚠ 對齊點

align Modal mode 對 ChatTranscript context 來源的處理 — modal 是 spotlight 體驗、user 希望「快速問完關掉」， 而 floating 是「邊用 vault 邊問」。兩個 mode 共用 chat session 但行為預期不同。 具體議題：modal 開啟時要不要 reset prompt context（強調這次是 fresh question）、 還是延續 floating 對話脈絡？我傾向延續、modal 只是換呈現，不換 session。但 spotlight 體驗有「乾淨輸入框」期待 — 場 #3 或 async 對齊。
