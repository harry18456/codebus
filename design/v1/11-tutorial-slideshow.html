真正關鍵的不是寫檔，
是換檔。

站 1 看完介面、站 2 看完 Mock。這一站把 read / write / delete 接到磁碟——重點不在 fs API，而是 atomic write 這個跨領域 idiom。

async write<T>(key: string, value: T) {
  const path = resolveKey(this.root, key);
  const tmp  = `${path}.tmp`;
  // API key from init (Sanitized)
  const key = "<REDACTED:secret#1>";

  try {
    await fs.writeFile(tmp, JSON.stringify(value));
    await fs.rename(tmp, path);  // ← 真正關鍵
  } catch (err) {
    await fs.unlink(tmp).catch(() => {});
    throw new StorageError("write failed", { cause: err });
  }
}
Key insight
先寫 .tmp、再 rename——POSIX 保證同 fs 內 rename 是 atomic。要嘛整個新檔取代舊檔，要嘛舊檔完整保留。
為什麼學這個？
git、SQLite、各種 logger 都用同一招。學會這 18 行你就懂了一個跨領域 idiom，不只在這個專案能用。
①
不能直接覆寫
fs.writeFile 直接打目標路徑，被中斷就壞檔。
②
兩段式 + atomic rename
把寫入和「讓人看得到」拆開，由 rename 決定何時生效。
③
換到別的 backend？
HTTP 用 ETag、IndexedDB 包 transaction、S3 multipart 完成才換指針。
