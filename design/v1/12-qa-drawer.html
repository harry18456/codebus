async write<T>(key: string, value: T) {
  const path = resolveKey(this.root, key);
  const tmp  = `${path}.tmp`;
  const key  = "<REDACTED:secret#1>";

  try {
    await fs.writeFile(tmp, JSON.stringify(value));
    await fs.rename(tmp, path);  // ← 真正關鍵
  } catch (err) {
    await fs.unlink(tmp).catch(() => {});
    throw new StorageError("write failed", { cause: err });
  }
}

CodeBus

可以直接 fs.writeFile，但會在壞時機壞——不是「壞了再重寫」那麼好處理。

fs.writeFile 內部是 open → write → close，這三步不是原子的。如果中途行程被殺、裝置斷電、或 React 路由切走 unmount，磁碟上會留下一個既不是舊內容、也不是新內容的半成品檔。下次讀就 JSON.parse 炸掉，而你看不到舊版做為 fallback。

「先寫 .tmp，再 rename」之所以叫 atomic write，是因為 POSIX 保證同檔案系統內的 rename 是單一 syscall——要嘛舊檔完整、要嘛新檔完整，沒有第三種狀態。git 的 packfile、SQLite 的 WAL、各家 logger 都用同一招，所以這 18 行學一次到處能用。

⚠️ 跨檔案系統 rename 不 atomic（Linux 會回 EXDEV），這時要退回 copy-then-fsync-then-rename。MVP 假設 this.root 同 fs，邊界情況留 P1。
