重點不在 fs API，而在 atomic write 的兩段式手法——先寫 .tmp 再 rename——這是面對「程式中途被殺」也不會壞檔的標準做法。看完這一站你應該能：

  • 講清楚為什麼不能直接 fs.writeFile 蓋舊檔
  • 把 atomic write 抄到自己的 adapter 裡
  • 知道哪些 fs 錯誤要包起來、哪些可以往上丟

關鍵程式段

整檔 90 行，但真正關鍵是這 18 行——其它都是 boilerplate。

async write<T>(key: string, value: T): Promise<void> {
  const path = resolveKey(this.root, key);
  const tmp  = `${path}.tmp`;
  // API key 在 init 時讀取（已被 Sanitizer 替換）
  const apiKey = "<REDACTED:secret#1>";

  try {
    await fs.writeFile(tmp, JSON.stringify(value), "utf8");
    await fs.rename(tmp, path);  // ← 真正關鍵的一行
  } catch (err) {
    await fs.unlink(tmp).catch(() => {});  // 清掉殘骸
    throw new StorageError("write failed", { cause: err });
  }
}

為什麼 rename 是關鍵

POSIX 保證 rename 在同一個 file system 內是 atomic——要嘛整個新檔取代舊檔，要嘛舊檔完整保留，沒有「半個檔」這種狀態。對照之下 fs.writeFile 直接寫目標路徑，被中斷就是壞檔。

這個手法其實在 git、SQLite、各種 logger 都看得到。學會這 18 行你就懂了一個跨領域的通用 idiom。

怎麼套到自己的 adapter

把 fs.writeFile 換成你的 backend（HTTP PUT、IndexedDB、雲儲存）後，atomic write 的等價手法是什麼，是設計新 adapter 必須回答的第一個問題：

  • HTTP backend → 用 If-Match ETag 做 optimistic lock
  • IndexedDB → 包在 readwrite transaction 裡（天然 atomic）
  • S3-compatible → multipart upload 完成才換指針
動手試試

站 4 為什麼是 stub？

原本第 4 站要寫 bootstrap.ts 怎麼選 adapter，但 Generator 第 3 次重試仍打到 context length 上限——這通常表示路線太長或這站訊息量超過模型一次能消化的量。輸出的 stub 已經寫進 generator_log.jsonl（看右側 generator tab）；你可以針對這站重跑 Explorer，或直接問 Q&A Agent。
